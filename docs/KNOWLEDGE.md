## 2.1 Vue -> React 对照学习索引
| Vue 经验 | React 对应思路 | 迁移提示 |
| --- | --- | --- |
| `ref/reactive` | `useState` | React 更强调不可变更新 |
| `computed` | `useMemo` | 只做性能优化，不承载副作用 |
| `watch/watchEffect` | `useEffect` | 重点是依赖数组与副作用时机 |
| `props + emit` | `props + callback` | 单向数据流一致 |
| `slot` | `children` / render props | React 组合模式更常见 |
| `pinia/vuex` | Zustand/Context | 全局状态只存跨页面共享数据 |
| `vue-router` 守卫 | 路由守卫组件/路由层拦截 | 常与权限系统联动 |
| `setup` 逻辑复用 | 自定义 hooks | hooks 是 React 复用核心 |

### 4.1 React 渲染与状态最小闭环
- 任务关联：W1-1 / W1-3
- 概念：UI 是 state 的映射，状态变化会触发重新渲染。
- 为什么需要：后台页面交互非常多，必须先理解“何时渲染、为什么渲染”。
- 常见坑：把临时变量当状态；在 render 中做副作用。
- 自测方式：能解释“点击按钮切主题时，哪些组件会重新渲染”。

### 4.2 路由驱动页面而不是页面写死
- 任务关联：W1-1
- 概念：页面由路由匹配决定，不应在 `App.tsx` 直接硬编码页面组件。
- 为什么需要：后续权限控制、面包屑、菜单高亮都依赖路由系统。
- 常见坑：路由和菜单双份 hardcode，后期维护成本很高。
- 自测方式：删除某个页面路由后，菜单和访问路径是否同时失效。

### 4.3 Layout + Outlet 的组合思想
- 任务关联：W1-2
- 概念：Layout 负责壳层，`Outlet` 负责注入当前子路由页面。
- 为什么需要：保持布局稳定、内容区可替换，是中后台标准结构。
- 常见坑：在 Layout 中直接写具体业务页面，导致无法复用。
- 自测方式：切换不同子页面时，侧边栏和头部保持不变，只有内容区变化。

### 4.4 配置驱动菜单
- 任务关联：W1-4
- 概念：用配置描述菜单结构与路由映射，而不是在组件里散写菜单项。
- 为什么需要：后续接后端动态菜单和权限过滤时可以平滑演进。
- 常见坑：菜单 key 与路由 path 不一致，造成高亮错误。
- 自测方式：新增一个路由配置后，菜单能自动展示并正确高亮。

### 4.5 页面状态与全局状态边界
- 任务关联：W1-3
- 概念：页面局部交互状态优先本地管理，全局状态只放跨页面信息。
- 为什么需要：避免 store 膨胀，提高可读性和调试效率。
- 常见坑：把表单临时字段放全局，造成状态污染。
- 自测方式：说出“主题、用户信息、表单输入”分别该放哪一层状态。

### 4.6 双路由来源模式（frontend/backend）
- 任务关联：W1-7 / T1-6
- 概念：系统支持“前端静态路由”与“后端动态路由”两种来源，并可切换。
- 为什么需要：适配不同项目阶段和团队协作方式。
- 在本项目中的使用位置：路由初始化层、菜单生成层、权限过滤层。
- 常见坑：两种来源路由的 meta 结构不一致，导致菜单高亮或权限判断错误。
- 自测方式：切换来源配置后，核心页面可访问且菜单与权限行为一致。

### 4.7 Zustand + React Query + Nitro 职责边界
- 任务关联：T2-3 / T2N-3
- 概念：
  - React Query：服务端状态缓存与同步。
  - Zustand：客户端全局状态。
  - Nitro：本地 mock/BFF 服务层。
- 为什么需要：把“状态管理”和“数据来源”分层，避免单一工具承担过多职责。
- 常见坑：把接口数据直接塞进 Zustand，导致缓存、重试、失效策略混乱。
- 自测方式：能说明“列表数据、主题状态、mock 接口”分别由谁负责。

### 4.8 Nitro 在本项目中的定位
- 任务关联：T2N-1 / T2N-2
- 概念：Nitro 在当前阶段作为 mock + BFF，后续可平滑替换真实后端。
- 为什么需要：当前没有后端支持，但又要演练真实中后台接口协作模式。
- 在本项目中的使用位置：`server/api/*`（接口）、中间件（鉴权模拟）、route rules（可选缓存）。
- 常见坑：mock 数据结构与真实后端契约差异太大，后期替换成本高。
- 自测方式：前端无需改动页面逻辑，只切换 baseURL 即可换数据源。

### 4.9 路由配置驱动菜单（Day 2）
- 任务关联：D2-1 / D2-2 / D2-3
- 概念：菜单不再硬编码，而是由路由配置（含 `meta`）生成。
- 为什么需要：新增/调整页面时只改路由配置，减少重复维护。
- 在本项目中的使用位置：`src/router/index.tsx` + `src/layouts/tt-sider/helper.ts`
- 常见坑：
  - 使用 router 实例而非路由配置数组做转换；
  - 菜单 key 与真实 path 不一致导致高亮错误；
  - 把调试日志留在 helper 中导致噪声。
- 自测方式：新增一个带 `meta.title` 的路由后，菜单自动出现且可跳转。

### 4.10 RouteObject 的 meta 扩展方式（Day 2）
- 任务关联：D2-5
- 概念：React Router 的 `RouteObject` 默认不包含 `meta`，需自定义路由类型扩展。
- 为什么需要：避免 TS 报错，并让菜单/权限字段具备静态类型约束。
- 在本项目中的使用位置：`src/router/types.ts` + `appRoutes as RouteObject[]`
- 常见坑：直接在 `RouteObject[]` 上写 `meta` 导致类型报错。
- 自测方式：路由中使用 `meta` 且 `pnpm build` 无 TS 报错。
