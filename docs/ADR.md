# 架构决策记录（ADR）

更新时间：2026-05-28

## 使用说明
- 每条 ADR 记录一个“重要、可争议”的技术决策。
- 结构：背景 -> 备选方案 -> 决策 -> 后果。
- 目标：让未来回看时知道“为什么这么做”，而不只是“现在是什么”。

---

## ADR-001 路由方案选择
- 日期：2026-05-28
- 状态：Accepted
- 背景：
  - 项目需要支持中后台复杂路由、嵌套路由、权限控制。
  - 你有 Vue Router 经验，需迁移到 React 生态。
- 备选：
  - A. React Router v7
  - B. TanStack Router
- 决策：
  - 采用 React Router v7。
- 原因：
  - 社区成熟、文档完整、与现有项目依赖一致。
  - 便于和 Ant Design Pro 风格后台实践对齐。
- 后果：
  - 需要明确 v7 包结构与导入方式。
  - 后续按 `frontend/backend` 双路由来源实现。

## ADR-002 状态管理分层
- 日期：2026-05-28
- 状态：Accepted
- 背景：
  - 项目既有全局 UI 状态，也有大量接口异步状态。
- 备选：
  - A. 全部放 Zustand
  - B. Zustand + React Query 分层
- 决策：
  - 采用 `Zustand + React Query` 分层。
- 原因：
  - 职责更清晰：Zustand 管本地全局状态，React Query 管服务端状态。
- 后果：
  - 需要约束边界，防止接口数据回流到 Zustand。

## ADR-003 无后端阶段的数据源策略
- 日期：2026-05-28
- 状态：Accepted
- 背景：
  - 当前没有后端服务，但需要真实联调体验。
- 备选：
  - A. 页面内写死 mock 数据
  - B. 引入 Nitro 作为本地 mock/BFF
- 决策：
  - 引入 Nitro。
- 原因：
  - 更接近企业真实协作模式（接口契约、鉴权、菜单权限模拟）。
  - 后续替换真实后端成本更低。
- 后果：
  - 需要维护 `API-CONTRACT.md` 与 mock 一致。
  - 需关注 Nitro 版本策略（稳定优先 + 前沿跟踪）。

## ADR-004 文档驱动的学习开发流程
- 日期：2026-05-28
- 状态：Accepted
- 背景：
  - 项目目标不仅是完成功能，还要系统学习与复盘。
- 备选：
  - A. 仅代码推进
  - B. 代码 + 多文档联动
- 决策：
  - 使用多文档联动（任务、知识、行业、Nitro、Git、元素、契约、ADR、发布）。
- 原因：
  - 降低遗忘和返工，提高可复用学习资产沉淀。
- 后果：
  - 每次迭代要同步更新文档，成本可控但收益高。
