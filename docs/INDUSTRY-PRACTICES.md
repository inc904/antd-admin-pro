# 中后台行业通用写法与设计思路

更新时间：2026-05-28

## 1. 文档目标
- 这不是某个框架专属文档，而是中后台前端在行业内较通用的设计方式。
- 目标是帮你建立“可迁移能力”：换公司、换技术栈也能用。
- 本文档会随着项目推进持续更新，不追求一次写完。

## 2. 中后台项目的核心设计原则

### 2.1 配置驱动优先
- 路由、菜单、权限、表格列、表单项尽量配置化。
- 价值：减少重复代码，便于动态化（后端下发菜单、权限点）。

### 2.2 分层清晰
- 推荐分层：`UI 组件层` -> `页面编排层` -> `业务逻辑层 hooks` -> `数据访问层 services/request`。
- 价值：让页面组件更“薄”，更好测试和复用。

### 2.3 约定优于随意
- 命名、目录、错误处理、接口返回处理方式要统一。
- 价值：多人协作效率更高，代码可读性更稳定。

### 2.4 可观测性优先于“看起来能跑”
- 明确 loading、error、empty、success 四种状态。
- 关键交互要可追踪（操作日志、埋点、错误上报）。

### 2.5 渐进增强
- 先做最小闭环，再做高级能力（如缓存策略、乐观更新、复杂权限）。
- 价值：降低风险，持续交付。

## 3. 行业内常见架构分工

### 3.1 路由与权限
- 路由控制“页面能否进入”，权限控制“按钮能否操作”。
- 行业内常见 RBAC：`用户 -> 角色 -> 权限点`。
- 菜单一般来自路由配置或后端菜单树，前端做二次过滤。
- 行业内常见做法：同时支持静态路由与动态路由，通常通过 `routeSource` 配置切换：
  - `frontend`：前端维护全量路由，适合快速开发和早期阶段。
  - `backend`：后端下发菜单/路由元信息，前端做映射与过滤，适合权限复杂系统。
- 工程建议：两种来源先 normalize 成统一的 `AppRoute` 结构，再喂给菜单与路由渲染层。

### 3.2 请求与数据层
- `request` 统一处理 token、错误码、重试、超时、取消请求。
- 页面不直接调用底层网络库，统一通过 `services`。
- React Query 常见写法：`query 读数据`，`mutation 改数据`，成功后失效相关 `query key`。
- 行业常见补充：当前端缺少后端支持时，引入轻量 BFF/mock 层（例如 Nitro）来模拟真实接口协作，而不是把 mock 数据直接写死在页面组件中。

### 3.3 状态管理边界
- 全局状态：用户信息、主题、权限、全局字典。
- 页面状态：筛选条件、弹窗开关、表单临时值。
- 行业内典型问题：把所有状态都丢进全局 store，导致维护困难。

### 3.4 表单与表格
- 表格要标准化：筛选、分页、排序、批量操作、导出。
- 表单要标准化：校验、默认值、提交态、回显、错误提示。
- 常见组合：`List + SearchForm + Table + ModalForm`。

### 3.5 异常与反馈
- 统一错误提示策略：业务错误提示可读信息，系统错误做兜底文案。
- 常见页面级异常：403/404/500。
- 关键操作使用二次确认和可恢复机制（例如软删除）。

### 3.6 Mock/BFF 的行业演进路径
- 第 1 阶段：静态 mock 数据（仅用于 UI 占位）。
- 第 2 阶段：本地 mock 服务（Nitro 路由，支持分页/筛选/权限模拟）。
- 第 3 阶段：BFF 层（聚合多个后端服务、统一鉴权与错误格式）。
- 第 4 阶段：替换为真实后端，前端保持服务层接口不变。
- 关键实践：从第 2 阶段开始就定义稳定的数据契约（DTO、分页模型、错误码）。

## 4. 交付质量的行业基线

### 4.1 工程规范
- TypeScript 开启严格模式（逐步增强）。
- ESLint + Prettier + 提交前校验（husky + lint-staged）。
- CI 至少包含 `lint + build`。

### 4.2 测试策略
- 工具函数和核心 hooks 做单元测试。
- 关键业务流程做最少量集成测试（登录、列表查询、新增提交流程）。

### 4.3 性能基线
- 路由懒加载、组件按需加载、长列表优化、请求缓存。
- 关注构建产物体积和首屏加载指标。

### 4.4 安全与合规
- 防 XSS：避免不安全 HTML 注入。
- 防越权：前端校验只做体验，权限以后端鉴权为准。
- 敏感信息不落地或加密落地（视公司规范）。

## 6. Vue 经验迁移到 React 的行业思路
- 迁移重点不是 API 对应，而是架构职责对应。
- 推荐迁移路径：
  - 从“语法迁移”转向“数据流迁移”（响应式思维 -> 单向数据流）。
  - 从“组件选项组织”转向“hooks 逻辑组织”。
  - 从“路由文件配置”转向“路由对象 + 组件组合 + 数据加载策略”。
- 团队协作常见要求：在 code review 中明确“这段实现是 Vue 习惯迁移，还是 React 原生模式”，减少混搭导致的维护成本。

## 7. Nitro 选型与版本策略（本项目）
- 官方站点已展示 Nitro v3 文档与能力；但官方仓库 README 明确你当前看到的是 v3 分支，稳定版仍需关注 v2 线。
- 本项目建议：
  - 学习路径：优先掌握通用概念（路由、handler、storage、cache），这些在主版本间可迁移。
  - 落地路径：项目实操优先选择稳定依赖版本；并记录 v3 差异点，后续再升级。
- 决策原则：学习可以前沿，项目依赖要稳。
